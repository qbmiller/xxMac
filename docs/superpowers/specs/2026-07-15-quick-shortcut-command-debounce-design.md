# 快捷指令命令脚本输入防抖设计

## 背景

启动器识别到命令脚本快捷指令后，会在每次输入变化时立即启动脚本并写入最近操作历史。连续输入 `yd Argentina` 时，`yd a`、`yd ar`、`yd arg` 等中间状态因此都会启动脚本并形成历史记录。

本次只增加命令脚本输入防抖，不处理最近操作的会话合并，也不处理方向键浏览历史时的输入框回填。

## 方案比较

1. 对启动器全部搜索统一防抖：实现集中，但会延迟应用搜索、计算器和网页快捷指令，不符合启动器即时反馈体验。
2. 只在命令脚本调度处防抖：保留其他搜索类型的即时响应，并直接阻止中间输入启动外部进程。采用此方案。
3. 启动脚本后再取消旧进程：可以处理已经启动的旧任务，但需要扩展进程生命周期管理，超出本次范围。

## 行为设计

- 命令脚本快捷指令匹配成功后等待 `400ms`，到期时只启动最后一次尚有效的输入。
- 防抖期间收到新的命令输入时，取消尚未启动的旧任务并重新计时。
- 输入切换为非命令、等待参数状态，或启动器关闭、切换模式时，取消尚未启动的任务。
- 最近操作历史在脚本真正开始执行时写入，不在防抖调度时写入。
- 已经越过防抖窗口并启动的脚本不强制终止；现有运行标识继续保证旧脚本晚返回时不能覆盖新输入的结果。
- 应用搜索、窗口命令、计算器、网页快捷指令、浏览器搜索、剪贴板和 Snippets 行为不变。

## 状态与数据流

1. 启动器输入变化并识别出命令脚本快捷指令。
2. ViewModel 取消上一次待执行任务，保存本次输入对应的运行标识。
3. `400ms` 内没有更新时，ViewModel 再次确认模式、输入和运行标识仍有效。
4. 写入本次快捷指令历史并异步启动脚本。
5. 脚本返回后沿用现有运行标识校验，只展示当前输入对应的输出。

## 测试与验收

- 快速连续提交多个命令输入时，防抖窗口内只有最后一次进入实际执行阶段。
- 待执行期间切换为非命令输入或重置启动器，不执行已经取消的命令。
- 历史写入与实际执行同时发生；被防抖取消的输入不写历史。
- 非命令搜索仍立即刷新。
- 完整测试套件和 Swift 构建通过。
- README 中现有“快捷指令命令脚本”和“最近操作历史”描述仍准确；本次不新增配置项或存储位置，因此“通用 > 配置目录”无需新增数据类别。
